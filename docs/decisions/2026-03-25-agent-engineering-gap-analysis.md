# 复盘：Agent 工程层提升 — 差距分析与下一步

## 日期
2026-03-25

## 背景

完成 Agent 工程层提升（11 项）和生态兼容（8 项）后，对 CCCLaw 与 Claude Code 的差距做一次诚实评估。

## 核心结论

**代码写完 ≠ 能力到位。** 11 项工程优化的代码全部落地，但大部分没有经过真实 Agent + LLM 的端到端验证。差距评估是主观判断，不是数据。

### 框架差距（CCCLaw vs CC，同模型下）

1. **Prompt 工程积累**：CC 的每个工具描述、system prompt 都经过百万用户实战打磨。我们写了分阶段 prompt 和 few-shot 示例，但没经过迭代优化
2. **并行工具调用**：CC 让 Opus 一轮并行 5+ 工具，我们的 agent.ts 还是串行逐个执行
3. **错误恢复质量**：CC 出错后的自我纠正链路成熟，我们有 Write-Verify-Fix 但反馈 prompt 很粗
4. **Context 管理**：CC 的压缩保留关键信息，我们加了相关性评分但没验证效果
5. **生态成熟度**：CC 的 MCP/Skill 经过大量用户使用打磨，我们刚通链路

## 经验教训

1. **先建度量再写代码**：评测基准应该第一个做，而不是最后一个。我们把评测框架搭了但没接真实 Agent，导致所有优化都是"我觉得有用"
2. **写完代码只是 50%**：工程优化的价值在于实际效果，不在于代码行数。11 项改进的代码完成度 100%，但验证度只有 ~30%
3. **框架打磨需要实战**：CC 的 prompt 不是一次写好的，是经过无数次失败迭代出来的。我们的 prompt 是一次性写的，大概率需要多轮迭代
